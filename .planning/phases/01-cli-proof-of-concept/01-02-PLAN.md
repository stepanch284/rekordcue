  <files>writer.py</files>
  <action>
Create `writer.py` in the project root with four functions:

1. `require_rekordbox_closed()`:
   - Import `psutil`.
   - Iterate `psutil.process_iter(['name'])`.
   - If any process name contains 'rekordbox' (case-insensitive), raise `RuntimeError` with the message: `"rekordbox.exe is running (PID {proc.pid}). Close Rekordbox before writing cue points."`.
   - Catch `psutil.NoSuchProcess` and `psutil.AccessDenied` inside the loop (process may vanish mid-iteration) — skip those processes silently.

2. `backup_master_db(db_path: Path) -> Path`:
   - Import `shutil`, `time`.
   - Generate timestamp: `time.strftime('%Y%m%d_%H%M%S')`.
   - Backup path: `db_path.parent / f'master.db.rekordcue_{timestamp}.bak'`.
   - Copy with `shutil.copy2(db_path, backup_path)`.
   - If copy raises any exception, re-raise with message "Backup failed — write aborted. No changes made."
   - Return backup_path.

3. `write_hot_cue(db, content, position_ms: int)`:
   - Import `uuid` and `from pyrekordbox.db6.tables import DjmdCue`.
   - Generate ID: `str(db.generate_unused_id(DjmdCue))`.
   - Create `DjmdCue` ORM object with these exact field values (per RESEARCH.md verified patterns):
     - `ID` = generated ID string
     - `ContentID` = `content.ID`
     - `ContentUUID` = `content.UUID`
     - `InMsec` = `round(position_ms)`
     - `InFrame` = 0
     - `InMpegFrame` = 0
     - `InMpegAbs` = 0
     - `OutMsec` = -1
     - `OutFrame` = -1
     - `OutMpegFrame` = -1
     - `OutMpegAbs` = -1
     - `Kind` = 1 (hot cue slot A)
     - `Color` = 255 (default hot cue color, NOT 0-8 palette index)
     - `ColorTableIndex` = 22 (default hot cue appearance)
     - `ActiveLoop` = 0
     - `Comment` = '' (empty string, no label — matches Rekordbox native behavior)
     - `BeatLoopSize` = -1
     - `CueMicrosec` = `round(position_ms) * 1000`
     - `UUID` = `str(uuid.uuid4())`
   - Do NOT set `created_at` or `updated_at` — the StatsFull mixin auto-sets them to `datetime.now`.
   - Call `db.add(cue)`.
   - Call `db.commit()` — this has a built-in rekordbox process guard and manages USN increments automatically.

4. `safe_write_sequence(db, content, position_ms: int) -> Path`:
   - Call `require_rekordbox_closed()` first (early, clean error before any mutation).
   - Compute `db_path = db.db_directory / 'master.db'`.
   - Call `backup = backup_master_db(db_path)` and print the backup path.
   - Call `write_hot_cue(db, content, position_ms)`.
   - Return backup path.
   - If `write_hot_cue` raises, the backup already exists for recovery. Print "Write failed — backup available at {backup}" and re-raise.

Use `from pathlib import Path` throughout. No raw sqlite3 usage (DB is encrypted).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| App -> master.db (WRITE) | Writing cue rows to the encrypted database via pyrekordbox ORM |
| App -> filesystem (backup) | Creating a copy of master.db before writes |
| CLI input -> track_id | User-supplied string used in DB query |
| App -> OS process list | Reading process names via psutil to detect Rekordbox |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | master.db corruption | mitigate | Pre-write backup via shutil.copy2(); psutil process guard ensures Rekordbox is closed; pyrekordbox commit() has built-in process guard as second layer; ORM handles transaction safety and USN management |
| T-01-02 | Tampering | Partial write / interrupted transaction | mitigate | pyrekordbox commit() uses SQLAlchemy session with implicit transaction; if commit fails, no rows are persisted. Backup file exists for manual recovery. |
| T-01-03 | Information Disclosure | AppData path in console output | accept | Path printed to user's own terminal for their own library; no network transmission; acceptable for CLI tool |
| T-01-04 | Denial of Service | Disk full during backup | mitigate | backup_master_db() re-raises copy failure with clear message "Backup failed — write aborted"; safe_write_sequence() calls backup BEFORE write, so no DB mutation occurs if backup fails |
| T-01-05 | Spoofing | Malicious track_id input | mitigate | track_id passed to SQLAlchemy ORM query (parameterized); no raw SQL string interpolation. Invalid ID returns None, caught by get_track() ValueError. |
| T-01-06 | Elevation of Privilege | Writing to DB owned by another user | accept | Tool runs as the current user; master.db is in the current user's %APPDATA%; no privilege escalation possible. File permissions are inherited from the user's profile. |
| T-01-07 | Tampering | Backup file overwrites existing backup | accept | Timestamp granularity is 1 second (YYYYMMDD_HHMMSS); two writes within the same second would overwrite. Acceptable risk for a CLI PoC — user would not run twice in 1 second. |
</threat_model>
